# Current Issues Report

## Problem: Spec Preview Modal Overflow and Height Issue
The spec preview modal is not handling overflow and height correctly. The content inside the modal is larger than the visible screen, but the modal body is not becoming properly scrollable. Because of this:
- Users cannot smoothly scroll through the full document
- Parts of the spec become inaccessible
- The modal content exceeds the available viewport height
- The scrolling behavior feels broken or stuck

**Root Cause**: The modal layout is failing because the document container (`<ScrollArea>`) was not configured as a proper scrollable area within a fixed-height modal. Specifically, the root `ScrollArea` component lacked an `overflow-hidden` constraint, and the padding on the modal's `ScrollArea` element interfered with the height calculation of the inner viewport.

## Resolution
1. **Added `overflow-hidden` to `ScrollArea`**: Modified `components/ui/scroll-area.tsx` to include `overflow-hidden` on the `ScrollAreaPrimitive.Root`. This allows the inner viewport to properly calculate when it should start scrolling.
2. **Fixed Padding in `SpecPreviewModal`**: Moved the `p-6` padding class from the `<ScrollArea>` root element to an inner `<div>` wrapping the markdown content in `components/editor/spec-preview-modal.tsx`. This ensures that the viewport's `size-full` utility accurately calculates the height without being offset by padding.
3. **Documentation**: Added code comments inline where these specific fixes were made to explain their necessity for future maintainability.

## Problem: AI Replaces Current Architecture Diagram
When users requested the AI to create another architecture diagram, the AI would overwrite the existing canvas instead of creating a new diagram separately. The previous architecture disappeared.

**Root Cause**: 
The issue had a two-part root cause:
1. **Liveblocks API Parsing Bug**: The Liveblocks REST API v2 returns CRDT objects (like `LiveObject` and `LiveMap`) wrapped in `{ liveblocksType: "...", data: {...} }` structures. The AI agent's code was expecting `storage.data.flow.nodes` to directly contain the nodes. Because of the wrapper, it evaluated as `undefined`. This had two catastrophic effects:
   - The AI was fed an empty `{}` state, making it blind to the existing diagram.
   - The agent mistakenly believed the `nodes` map was missing from storage and issued a JSON patch `{ op: "add", path: "/flow/nodes", value: {} }` which completely erased the existing nodes from the Liveblocks room before adding the new ones.
2. **Prompt Instructions**: The AI lacked explicit instructions to preserve existing nodes and generated predictable IDs which caused collisions.

## Resolution
1. **Fixed Liveblocks Storage Parsing (`trigger/design-agent.ts`)**: Updated the data extraction logic to properly unbox the `data` properties from `flowObj` and `nodesObj`. The script now accurately detects existing nodes, preventing the destructive `{ op: "add", path: "/flow/nodes", value: {} }` patch. The AI also now correctly "sees" the current canvas state.
2. **Prompt Constraints Updated**: Modified the system prompt for the AI to explicitly restrict overwriting or deleting existing nodes. Instructed it to use strictly unique IDs (with random numbers) for new nodes.
3. **Programmatic Safeguard**: Added logic in the action parser that explicitly skips any `deleteNode` or `deleteEdge` actions generated by the AI unless the user's prompt explicitly contained destructive keywords (`delete`, `remove`, `clear`, etc.).
